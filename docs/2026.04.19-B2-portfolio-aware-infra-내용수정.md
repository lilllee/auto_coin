## 설계 요약

### 1. `portfolio_backtest` — 엔진

**전략 독립적 (Strategy-agnostic)**: `PortfolioSignal` 이 `(sliced_candles, date, ctx)` 를
받아 `dict[ticker, weight]` 를 반환한다. CSMOM / RCDB / XS-LOWVOL 등 어느 전략이든
이 인터페이스를 구현하면 같은 엔진에서 검증된다.

**실행 모델**:
1. Universe 날짜 교집합을 구한다 (`_align_universe`)
2. 초기 cash 로 시작, benchmark 는 universe equal-weight B&H 설정
3. 각 날짜 loop:
   - `i % rebal_days == 0` → signal 호출 → target_weights → `risk_budget` 상한으로 scale → `_apply_rebalance` (sell → buy 순서)
   - 매 날짜 mark-to-market 으로 equity/benchmark curve 갱신
4. 결과: equity_curve · benchmark_curve · trades · rebalance_events + 집계 메트릭

**Fee / slippage**: 매수·매도 양쪽 대칭 적용. `effective_price = close × (1±slip) × (1+fee)`.

### 2. `portfolio_walk_forward` — 검증

**SignalFactory 패턴**: `params → (PortfolioSignal, context_overrides)`. CSMOM 구현이
factory 를 제공하면 WF 가 train grid search → test 평가를 반복.

**집계 메트릭** (PLAN_CSMOM §4 Stage 4 기준 직결):
- `avg_test_return` / `avg_test_benchmark` / `avg_test_excess`
- `positive_excess_ratio` (excess > 0 윈도우 비율)
- `train_test_ratio` (`|avg_train| / |avg_test|` — 과적합 신호)
- `best_param_stability` (가장 빈번한 best_params 점유율)
- `total_test_trades` (test 합계 trade 수)

PLAN_CSMOM Stage 4 hard-fail / soft-warning / pass 기준을 이 지표들로 직접 판정 가능.

### 3. DailySnapshot 확장

기존 legacy 컬럼 유지 + 3개 nullable 컬럼 추가:

```python
portfolio_equity_krw: float | None        # 포트폴리오 평가액 (cash + MTM 포지션)
portfolio_excess_vs_bnh: float | None     # universe B&H 대비 누적 excess
active_strategy_group: str | None         # "legacy_single_ticker" | "csmom_v1" | ...
```

- Legacy 레코드는 이 컬럼에 NULL 저장 → 기존 KPI 로직 무영향
- CSMOM 활성화 시 write 경로가 이 3개를 채워 메인 KPI 지표로 사용

### 4. Sizing 입력 구조 (config.Settings)

```python
active_strategy_group: str = "legacy_single_ticker"
risk_budget_krw: float | None = None
atr_window_for_sizing: int = 20
portfolio_rebal_days: int = 7
```

**지금은 DB 매핑 없음** — pydantic Settings 에만 존재. CSMOM 구현 시점에 웹 settings UI
+ AppSettings 컬럼 + settings_service 매핑을 추가하면 됨. 지금은 "받을 자리" 만 마련.
