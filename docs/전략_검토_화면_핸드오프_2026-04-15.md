   - summary / rows / events 반환

3. **Step 3 — web router + API**
   - `GET /review`
   - `GET /review/data/{ticker}`
   - 최소 placeholder 템플릿 연결

4. **Step 4 — interactive UI**
   - `review.html`을 인터랙티브 리플레이 화면으로 확장
   - summary 카드
   - line chart + BUY/SELL marker
   - step slider
   - prev/next
   - 상세 패널
   - events / rows 테이블
   - loading / error / empty state

5. **후속 수정**
   - 검토 가능 최대 기간을 **5일 → 90일**로 확대

## 4. 현재 simulator 동작 방식

현재 simulator는 **전략 신호(strategy signal) 중심 replay**다.

정확히는:

- `fetch_daily(...)`로 기간+warmup 데이터 조회
- `create_strategy(...)`로 현재 전략 생성
- 각 row마다:
  - `MarketSnapshot(df.loc[:ts], current_price=close, has_position=...)`
  - `strategy.generate_signal(...)`
- BUY/SELL/HOLD를 기준으로 내부 position 상태만 갱신
- 결과:
  - `summary`
  - `rows`
  - `events`

### 중요한 제한

현재 simulator는 아래를 **반영하지 않는다**.

- `RiskManager`
- 운영 레벨 손절
- 스케줄러 기반 time exit
- 실제 주문/체결 상태

즉, 이 화면은 현재 기준으로 **“전략 신호 검토기”**에 가깝다.

## 5. 현재 보이는 한계 / 이미 확인된 포인트

실제 사용 중 확인된 중요한 포인트:

- 일부 전략은 `BUY` 이후 `HOLD`가 반복되어
  **entry-only replay처럼 보일 수 있음**
- 예:
  - `price<target`
  - `price>=target and price>ma5`
  - `already in position`

이 현상은 버그라기보다 현재 설계 특성 때문이다.

### 이유

예를 들어 `volatility_breakout` 계열은 전략 클래스 자체가 **진입 조건 중심**이고,
청산은 운영 계층(시간청산, 손절, 리스크 레이어)에서 처리되는 구조가 많다.

그래서 review simulator가 현재처럼 `strategy.generate_signal()`만 replay하면:

- 1회 BUY
- 이후 `has_position=True`
- 전략은 추가 BUY 안 하고 `HOLD`
- 운영 청산은 반영 안 함

결과적으로 **“buy 1번 + hold 반복”**처럼 보일 수 있다.

이건 다음 작업에서 개선 판단이 필요하다.

## 9. 개인적 추천(최소 수정 기준)

다음 단계에서 가장 실용적인 최소 수정 방향은:

> **review를 “전략 신호 검토기”로 유지하되, UI와 문구에서 명확히 표시하고, 옵션으로 운영 청산 반영 모드를 추가하는 것**

이유:

- 현재 구조를 거의 안 깨고 갈 수 있음
- 전략 신호만 보고 싶을 때와
  운영 청산까지 보고 싶을 때를 분리 가능
- 바로 RiskManager/실행 레이어를 기본 동작에 섞으면
  review 결과 의미가 흐려질 수 있음

즉, 기본값은 strategy-only 유지,
추가 옵션으로 operational exits를 붙이는 방향이 안전하다.
