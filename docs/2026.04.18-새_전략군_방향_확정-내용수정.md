| F6 | Window 당 샘플 사이즈 부족 — 30d test에 1 trade → 통계 파워 0 |
| F7 | 수수료 민감도 — expectancy 0.04~0.24%라 slip 0.05→0.15 만으로 edge 소멸 |

## 5. 후보 5개 평가

| 후보 | 핵심 | 순위 | 근거 |
|---|---|---|---|
| **A. CSMOM** (Cross-Sectional Momentum Rotation) | rank 기반 Top-K 로테이션 | **1** | threshold-free, universe 단일 파라미터, trade count 자동 확보, 인프라 70% 재사용 |
| **C. RCDB** (Regime-Conditioned Dip Buy) | BTC risk-on 레짐 하에서 BTC/ETH dip-buy | **2 (병렬)** | 단일자산 · 구현 단순 · CSMOM과 다른 edge 축(mean-reversion) |
| B. VTT (Volatility-Targeted Trend) | EMA cross + Chandelier trailing | 3 | trade count 부족 리스크 |
| D. XS-LOWVOL | 저변동성 cross-sectional | 4 | 구현은 쉬우나 edge 속도 느릴 가능성 |
| E. 1H-MR | 1시간봉 mean reversion | 5 | 가장 큰 alpha 잠재, 가장 큰 인프라 비용 |

**최종 채택**: 1순위 **CSMOM**, 2순위 **RCDB** (병렬 설계, CSMOM Stage 2 통과 후 착수)

## 6. 검증 기준 (Hard fail · Soft warning · Pass target 분리)

사용자 피드백 반영 — Stage 4 기준을 3단계로 분리. 이전 초안의 단일 threshold가 초기 필터로는 빡셀 수 있다는 지적 수용.

### Stage 4 (Walk-forward) 기준

| 지표 | Hard fail | Soft warning | Pass target |
|---|---|---|---|
| Avg test excess vs B&H | ≤ 0% | 0 ~ +1% | ≥ +2% |
| Positive excess window ratio | < 40% | 40 ~ 55% | ≥ 55% |
| Train/Test 배수 | > 5x | 3x ~ 5x | ≤ 3x |
| Test trade count 합계 | < 30 | 30 ~ 50 | ≥ 50 |
| Best-param 안정성 | < 30% | 30 ~ 60% | ≥ 60% |

- **Hard fail**: 즉시 전략 재설계 또는 폐기
- **Soft warning**: 다음 Stage 진입 허용, 단 결과 해석에 보수적 할인
- **Pass target**: 권장 기준, live 후보 적합

전체 Stage 1~7 세부 기준은 `docs/v4/PLAN_CSMOM.md` 참조.

## 7. 중요 인사이트 (사용자 피드백 반영)

### 7-1. CSMOM은 "전략 추가"가 아니라 "포트폴리오 엔진 확장"

사용자 지적 수용:

> CSMOM은 strategy 하나 추가가 아니라 백테스터/워크포워드/봇 구조를 portfolio-aware 하게 확장하는 작업

반영:
- 체크리스트 B2 (portfolio-aware 인프라) 를 B3 (CSMOM Stage 실행) 선행 조건으로 명시
- PLAN_CSMOM §3 에서 재사용/확장/폐기 계층 명시
- PLAN_CSMOM §5 에서 P1~P4 가 구현 필수, P5 이후는 gate 통과 후에만 진행

### 7-2. RCDB 2순위 의의

사용자 지적 수용:

> CSMOM = portfolio / cross-sectional 방향 · RCDB = simpler single-asset regime-conditioned direction. 병렬로 보면 좋다

반영:
- 체크리스트 B4 에 RCDB 설계 문서(`docs/v4/PLAN_RCDB.md`) 작성 항목 추가
- 착수 시점: CSMOM Stage 2 통과 이후 (hard-fail 시 RCDB로 빠르게 pivot 가능하도록)

### 7-3. Hard fail / Soft warning 분리

사용자 지적 수용:

> 운영상으로는 hard fail 기준과 soft warning 기준을 나누는 것도 괜찮다

반영:
- Stage 2 · Stage 3 · Stage 4 · Stage 6 모두 3단계 (Hard fail / Soft warning / Pass target) 로 재정의
- PLAN_CSMOM §4 전체에 반영
