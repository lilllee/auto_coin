# 오늘 작업 정리 (세션 초기화 이후)

기준 시작점:
> "지금 프롬포트 세션이 초기화 되서 ai가 기억을 못할꺼야  
> `RCDB 1차 설계/구현 프롬프트` 이 작업전 ai 가 프로젝트 이해할수 있게 가이드도 해줘야함"

---

## 1. 작업 목적

세션이 초기화되어 이전 맥락을 AI가 모르는 상태를 전제로,
아래 목표로 작업을 이어감.

1. **프로젝트 배경/검증 이력/현재 상태를 AI가 이해할 수 있게 self-contained 가이드 제공**
2. **RCDB(Regime-Conditioned Dip Buy) 트랙을 새 후보 전략으로 검증**
3. 기존 일봉 중심 검증 구조가 실시간/장중 전략과 안 맞는 문제를 인식하고,
   이후 **1H 실시간 전략 검증 구조로 전환하는 방향**까지 정리

## 2. 세션 초기화 대응: 프로젝트 이해 가이드 + RCDB 1차 프롬프트 작성

### 수행 내용
- 새 세션 AI가 프로젝트를 처음 보는 상태를 가정하고,
  아래 내용을 포함한 **프로젝트 이해 가이드 + RCDB 1차 설계/구현 프롬프트** 작성
  - 기존 전략군 탈락 이유
  - CSMOM 진행 결과 및 STOP 판정
  - 현재 재사용 가능한 운영/검증 인프라
  - RCDB의 목적과 철학
  - 이번 턴의 작업 범위(설계 + 최소 구현 + Stage 1/2 검증)

### 의도
- 단순히 전략 구현 지시만 하는 게 아니라,
  **왜 RCDB를 보는지 / 어떤 실패를 피해야 하는지 / 어떤 범위까지만 해야 하는지**
  를 새 AI가 오해 없이 이해하도록 만드는 것

## 3. RCDB v1 1차 구현 및 검증

### 구현/연결된 내용
- `rcdb` 전략 신규 구현
- 전략 registry 연결
- external regime ticker 지원
- generic backtest에서 strategy-defined exit hook 지원
- default UI 노출은 숨김 처리로 premature exposure 방지

### RCDB v1 철학
- BTC regime ON일 때만 진입
- 단기 dip + RSI 과매도 시 진입
- 청산:
  - `rcdb_trailing_exit`
  - `rcdb_time_exit`
  - `rcdb_regime_off`

### Stage 2 결과 요약
#### strict spec
- 예: `-8% / RSI30`
- BTC 1회, ETH 1회 수준으로 **너무 sparse**

#### 완화형/bounded candidate
- trade 수는 늘었지만:
  - **BTC는 B&H 대비 excess 음수**
  - **ETH 편향**
  - **exit 대부분이 time_exit**
- 결론:
  - 구조 자체는 연구 가치가 있으나
  - 지금 바로 walk-forward로 넘길 단계는 아님

### 당시 판정
- **REVISE**
- 이유:
  - strict하면 통계 부족
  - 완화하면 oversold thesis 약화 + ETH 편향
  - 한 번 더 구조/파라미터 조정 필요

## 4. RCDB v1.1 재설계

### 핵심 변경
- ATR trailing anchor:
  - `highest_close` → `highest_high`

### 의도
- trailing을 늦은 종가 기준 청산이 아니라
- **반등 후 give-back 방어용 구조**로 바꾸기

### 탐색 축
- `dip_threshold_pct`
- `rsi_threshold`
- `max_hold_days`
- `atr_trailing_mult`

### 결과 해석
- strict-middle 후보는 여전히 sparse
- balanced 후보는 BTC 열세 폭이 줄었지만 ETH 절대수익/expectancy 약화
- practical 후보는 trade 수는 충분하지만 thesis 희석 우려
- exit contribution은 보이기 시작했으나,
  **수익 중심은 여전히 time_exit**

### 당시 판정
- **HOLD**
- 이유:
  - STOP은 아님 (exit 구조가 일부 살아남)
  - PASS도 아님 (BTC excess 여전히 음수, ETH 편향 지속)
  - 한 번 더 bounded 조정할 가치가 있는 상태

---

## 5. RCDB v1.2 bounded redesign

### 내 판단에 따라 다음 방향 설정
- 단순 미세조정이 아니라,
  **BTC 일반화 + time_exit 의존도 축소**를 중심 목표로 한 bounded redesign 프롬프트 작성

### 재설계 의도
- strict vs relaxed 사이의 usable middle band 탐색
- exit reason 분해
- BTC/ETH 공통 후보 가능성 재평가

### 탐색 축
- `dip_threshold_pct`
- `rsi_threshold`
- `max_hold_days`
- `atr_trailing_mult`
- bounded range 안에서만 탐색

### 결과 해석
- usable middle band는 보였음
- BTC excess 음수 폭은 일부 감소
- 하지만:
  - **BTC는 여전히 B&H 열세**
  - **ETH 편향 지속**
  - **수익 핵심은 여전히 time_exit**
  - trailing/regime exit는 보조 역할

### 당시 판정
- **REVISE**
- 이유:
  - walk-forward로 넘기기엔 이르다
  - 더 같은 축의 숫자 튜닝보다는 **구조 재설계가 필요**하다는 쪽으로 방향 전환

---

## 6. RCDB v2 구조 재설계 방향 확정

### 핵심 판단
`v1 → v1.1 → v1.2` 결과를 바탕으로,
문제는 이제 threshold 숫자보다 **전략 구조 자체**에 가깝다고 판단.

### RCDB v1.x의 구조적 병목으로 정리한 것
1. strict하면 sparse
2. relaxed하면 ETH 편향
3. trailing exit가 수익 엔진이 아님
4. BTC가 benchmark를 못 이김
5. 전략이 사실상 `oversold buy 후 N일 보유`로 수렴

### RCDB v2 설계 원칙
- 절대 dip% 폐기/약화
- **vol-normalized dip**
- **reversal confirmation**
- **reversion exit**
- time_exit는 보조화
- BTC 일반화 우선
- walk-forward 가능한 최소 파라미터 유지

### 후보 구조 비교
- 후보 1: **Vol-Normalized Dip + Reversal Confirmation** ← 1순위
- 후보 2: Multi-Trigger Oversold State Machine
- 후보 3: Regime-Conditioned Ladder Entry

### 1순위로 고른 이유
- ETH 편향 원인이던 절대 dip% 문제를 바로 해결
- reversal confirm으로 falling knife 문제를 줄일 수 있음
- ladder/state-machine보다 단순하고 검증 친화적
- walk-forward에 실기 쉬움

### 구현 대상으로 정한 구조
**RCDB v2 = vol-normalized dip + reversal confirmation + reversion exit**

### 새 세션 AI용 self-contained 프롬프트에 포함한 내용
- 프로젝트 요약
- 기존 전략군 탈락 요약
- CSMOM STOP 결과
- RCDB v1.x 진행 결과
- 이번 턴의 목적:
  - `rcdb_v2.py` 최소 구현
  - registry 연결
  - enrich/backtest 연결
  - Stage 2 in-sample 검증
- 아직 하지 말 것:
  - walk-forward
  - live bot 연결
  - UI 연결

### v2 최소 구조 요구
- regime filter
- vol-normalized dip

- reversal confirmation
- reversion exit

- time_exit는 safety only

## 8. RCDB v2 1차 결과 해석

### 결과에 대한 해석
- **좋아진 부분**
  - time_exit 의존도가 크게 줄어듦
  - reversion exit가 실제 수익 실현 역할을 하려는 조짐
  - v1.x보다 “N일 보유 전략” 느낌은 약해짐

- **여전히 안 된 부분**
  - **BTC 일반화 실패**
  - 공통 후보로 보기 어려움
  - 결국 여전히 ETH 쪽에 더 유리
  - RCDB v2도 현 구조로는 운영 후보라고 말하긴 어려움

### 당시 판정
- **REVISE**
- 해석:
  - exit 구조는 개선됐지만
  - 공통 후보 전략으로는 부족
  - 같은 축 미세조정보다 **구조/타임프레임 자체를 다시 봐야 한다**는 쪽으로 생각이 이동

## 9. 중요한 방향 전환: 현재 테스트는 결국 일봉 기반이라는 점 확인

### 사용자 질문
- “근데 어짜피 현재 테스트 일봉으로 하는건가?”

### 답변/판단
- **그렇다. 지금까지 전략 검증은 사실상 일봉 기반**
- RCDB/기존 전략 모두
  - 일봉 수익률
  - 일봉 RSI/변동성
  - 며칠 단위 hold
  로 돌아가는 구조였음

### 여기서 나온 핵심 인식
- 사용자가 원하는 것은 **실시간/장중 판단 전략**
- 하지만 현재 검증 프레임은 **daily swing 전략**에 가까움
- 이 간극 때문에:
  - strict하면 sparse
  - 완화하면 thesis 희석
  문제가 반복될 가능성이 큼

즉:
> 문제는 전략 하나가 아니라,  
> **검증 타임프레임 자체가 실시간 전략 욕구와 안 맞는다**는 점

## 10. 다음 방향: 일봉 중심 auto_coin을 1H 실시간 전략 검증 구조로 전환

### 새 프롬프트 작성 목적
- 새로운 전략을 하나 더 바로 만들기보다,
  **프로젝트를 1H 전략 검증이 가능한 구조로 전환하는 설계**
  를 먼저 하도록 만드는 것

### 프롬프트에 담은 핵심 내용
1. 현재 일봉 구조의 한계 구조화
2. 왜 1H 전환이 필요한지
3. 1H 전환 설계 원칙
4. 데이터 / 백테스트 / walk-forward / 운영 계층별 설계안
5. 가장 먼저 구현할 범위 정의
6. 1H에서 먼저 실험할 전략 후보 추천

### 방향성 요약
- 완전 tick/5분봉부터 시작하지 않고
- **daily regime + 1H entry/exit** 구조를 1차 목표로 설정
- 목적:
  - 실시간 판단 전략과 검증 프레임의 정합성 개선
  - mean reversion / dip-buy 류 전략의 thesis 왜곡 감소
  - trade count 확보
  - walk-forward 통계력 개선

## 11. 전체 흐름 요약

### 초기 목표
- 세션 초기화 후 새 AI가 프로젝트를 이해하게 만들기
- RCDB를 새 후보 전략군으로 검증

### 실제로 진행된 흐름
1. 프로젝트 이해 가이드 + RCDB 1차 프롬프트 작성
2. RCDB v1 구현/검증 → **REVISE**
3. RCDB v1.1 재설계 → **HOLD**
4. RCDB v1.2 bounded redesign → **REVISE**
5. RCDB v2 구조 재설계 방향 확정
6. RCDB v2 최소 구현 프롬프트 작성
7. RCDB v2 결과 해석 → exit는 개선, 공통 후보는 여전히 부족
8. “현재 테스트는 결국 일봉”이라는 문제 인식
9. **1H 실시간 전략 검증 구조 전환 프롬프트 작성**

## 12. 현재 시점 최종 상태

### 확정된 것
- 기존 전략군: 운영 후보 탈락
- CSMOM: 현 구조 STOP
- RCDB v1.x: 숫자 미세조정만으로는 한계
- RCDB v2: exit 구조 개선은 있었지만 공통 후보로는 부족
- 현재 검증 프레임: **사실상 일봉 기반**
- 사용자가 원하는 전략 방향: **실시간/장중 판단 전략**

### 그래서 현재 가장 중요한 다음 단계
> **전략 하나 더 고르기보다,  
> auto_coin을 1H 실시간 전략 검증 구조로 전환하는 설계를 먼저 여는 것**

으로 정리할 수 있다.
